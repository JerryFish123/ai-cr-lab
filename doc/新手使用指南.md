## 3. 它怎么工作

### 流水线

```text
业务仓库：改代码 → Push / 开 PR
        ↓
GitHub 打电话到服务器（Webhook）
        ↓
服务器拉 diff（改了哪些行）
        ↓
发给大模型（火山方舟）
        ↓
写回 PR 评论 + 记入看板
        ↓
你在评论区 / 5002 看到结果
```

不需要手动点「开始审查」。**有变动就自动跑。**

### 两个仓库

| 仓库 | 干什么 |
|------|--------|
| **ai-cr-lab** | 工具本身。推 `publish` → 更新服务器上的程序 |
| **业务项目 A** | 你真正写的产品。**在这里挂 Webhook** |

两者可以完全分开。

## 5. 接上业务项目

### 接之前确认

1. http://39.96.125.66:5001/ 能打开  
2. 阿里云安全组放行了 **5001**（建议也放行 **5002**）  
3. `GITHUB_ACCESS_TOKEN` 对该项目能：读代码、写 PR 评论  

### 第 1 步：加 Webhook

路径：项目 A → **Settings** → **Webhooks** → **Add webhook**

| 项 | 填什么 |
|----|--------|
| Payload URL | `http://39.96.125.66:5001/review/webhook` |
| Content type | `application/json` |
| Secret | 可空 |
| Which events | 只勾 **Pull requests**、**Pushes** |
| Active | 勾选 |

保存后若显示 *never been triggered*：**正常**，还没事件。

### 第 2 步：触发一次（推荐开 PR）

1. 新建分支  
2. 随便改一行  
3. Push  
4. 创建 Pull Request  

### 第 3 步：按顺序验收

| 顺序 | 看哪里 | 成功长什么样 |
|------|--------|--------------|
| ① | Webhook → Recent Deliveries | 绿色 / **200** |
| ② | PR 评论（等 1～3 分钟） | 出现 Auto Review Result |
| ③ | http://39.96.125.66:5002/ | 多一条记录 |

三步都过 → 接好了。

## 7. 自动部署是什么

```text
push → publish
  → GitHub Actions
  → SSH 登录服务器
  → 写入 APP_ENV_FILE 到 conf/.env
  → 拉取最新代码
  → docker 重建
  → 5001 / 5002 变成新版本
```

### Secrets（敏感，知道名字即可）

| 名称 | 作用 |
|------|------|
| `ECS_HOST` | 服务器 IP |
| `ECS_USER` | SSH 用户 |
| `ECS_SSH_KEY` | 部署私钥 |
| `APP_ENV_FILE` | **整份** `.env`（Key、ep、Token 都在这） |

### 三条铁律

1. Secret 点编辑时 Value **是空的** → 正常，不是丢了  
2. **空着 Value 不要点 Update** → 会覆盖成空  
3. 大模型 Key / ep **没有单独 Secret**，都在 `APP_ENV_FILE` 里

## 8. 密钥怎么改

敏感信息**永远不要**提交进 Git。

| 位置 | 角色 |
|------|------|
| 本地 `conf/.env` | 你改配置的源头 |
| GitHub `APP_ENV_FILE` | 部署时同步用 |
| 服务器 `conf/.env` | 程序真正读取 |

### 换 Key / ep / Token

1. 改本地 `conf/.env`  
2. 整份更新到 Secret `APP_ENV_FILE`  
3. `git push origin publish`（或手动跑 Deploy）  
4. 确认还能正常审查 / LLM 连通  

### 常用配置名

| 名字 | 意思 |
|------|------|
| `OPENAI_API_KEY` | 大模型密码 |
| `OPENAI_API_MODEL` | 通常是 `ep-...` |
| `OPENAI_API_BASE_URL` | 方舟地址 |
| `GITHUB_ACCESS_TOKEN` | 读 diff、写评论 |
| `PUSH_REVIEW_ENABLED` | `1` 审 push；`0` 主要靠 PR |
| `REVIEW_STRATEGY` | `diff_only` 省钱；`agentic` 更强更贵 |

## 9. 模型、用量、花钱

### ep 去哪看

- 含义：推理接入点 ID  
- 控制台：**老版本** → 在线推理 / 推理接入点  
- 新版 Coding Plan 左侧常常没有这项 → **切老版**  
- 工具实际用的：看 `OPENAI_API_MODEL`

### 用量去哪看

| 页面 | 适合看 |
|------|--------|
| **Usage** | 一共用了多少 tokens、几次调用 |
| **Monitoring** | 刚才每分钟烧得多猛（TPM） |

### 钱大概多少

审查报告输出长 → 钱主要花在 **Output**。  
一次完整审查常见量级：**几分～一两毛人民币**（以账单为准）。

省钱：用 `diff_only`；Webhook 别勾多余事件；避免超大 PR。

## 11. 出问题怎么办

按「先外后内」查：

| 现象 | 先做什么 |
|------|----------|
| never been triggered | 先 Push 或开一次 PR |
| Delivery 红 / 超时 | 查 5001 能否打开 → 安全组 5001 → 容器是否在跑 |
| Delivery 绿但没评论 | 等 1～3 分钟 → 看看板 → Token 权限 → 大模型是否失败 |
| 只想审 PR | Webhook 只勾 Pull requests；或 `PUSH_REVIEW_ENABLED=0` 后重新部署 |
| 要审多个仓库 | 每个仓库各加一条同样的 Webhook |
| Secret 编辑是空的 | 正常；要改就整份粘贴覆盖 |
| Actions 失败 | 看日志；常见是服务器临时拉不到 GitHub，可重试 |
| 页面打不开 | 5001 → 安全组 → ECS 是否开机 |

## 12. 自测题

先写答案，再点开核对。

1. Webhook 配在工具仓库还是业务仓库？  
2. 收 GitHub 通知的是 5001 还是 5002？  
3. 大模型 Key 和 ep 在哪个 Secret？  
4. 改工具上线推哪个分支？  
5. Delivery 绿了但没评论，先查哪三步？  

<details>
<summary>参考答案</summary>

1. **业务仓库**  
2. **5001**（`/review/webhook`）  
3. **`APP_ENV_FILE`**  
4. **`publish`**  
5. 再等一会儿 → 看板有无记录 → Token / 大模型是否失败

> 改工具：本地改完 → 你推 `publish` → Actions 自动部署。
