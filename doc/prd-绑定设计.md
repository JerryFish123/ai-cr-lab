# PRD 如何挂进 PR（创建页方案）

## 结论（先看这个）

**可以，而且就应该用你截图里的「创建 Pull Request」页。**

| 人要提供的东西 | 放哪 |
|----------------|------|
| PRD 文件（PDF / Word） | 描述框下方 **Paste, drop, or click to add files** |
| 文字说明（范围第几到第几 + 补充） | 上面的 **Add a description** |

创建 PR（或之后编辑描述）保存后：

1. GitHub 把附件变成描述里的链接（常见为 `user-attachments` 或文件链接）  
2. Webhook 触发审查  
3. 我们的服务读 **PR 描述全文** → 解析范围/说明 → 下载附件 → 抽文本 → 和 diff 一起喂大模型  

**不需要**另做上传网站；第一版用人人都会的 GitHub 原生能力即可。

---

## 开发同学怎么填（标准模板）

在 **Add a description** 里直接粘贴下面这块（中文即可）：

```markdown
## 需求依据（给 AI Review 用，请保留标题）

- **PRD文件**: （把 PDF/Word 拖到下方附件区即可，会自动出现链接）
- **范围**: 3.2-3.5
- **说明**: 本次只做退款成功/失败状态流转；不做发票与对账

### 本次需求摘录（可选但强烈建议）
> 把 PRD 里相关段落复制 200～800 字到这里。
> 即使 PDF 解析失败，也能做业务比对。
```

### 字段约定

| 字段 | 必填？ | 写法示例 | 含义 |
|------|--------|----------|------|
| PRD文件 | 建议有 | 拖拽上传后出现的链接 | 需求原文 |
| 范围 | 建议有 | `3.2-3.5` / `p12-p15` / `§4.1-§4.3` | 只要这些段落 |
| 说明 | 可选 | 一句话边界 | 本次做/不做 |
| 本次需求摘录 | 强烈建议 | 粘贴关键段落 | 解析失败时的兜底 |

### 范围单位（第一版先定一种主写法）

优先支持：

1. **章节号**：`3.2-3.5`（按标题匹配）  
2. **页码**：`p12-p15`（PDF 较稳）  
3. **摘录兜底**：人直接粘贴，不依赖自动切片  

第一版：**一份 PR 只绑一份主 PRD**，降低复杂度。

## 和创建页 UI 的对应关系（你的参考图）

```text
┌ Add a title ─────────────────────┐
│  update                          │  ← 普通标题，给人不给机器
└──────────────────────────────────┘

┌ Add a description ───────────────┐
│  ## 需求依据 ...                 │  ← 范围 / 说明 / 摘录写这里
│  - **范围**: 3.2-3.5             │
│  - **说明**: ...                 │
│  （上传后这里还会出现文件链接）    │
│                                  │
│  📎 Paste, drop, or click...     │  ← PDF/Word 拖这里
└──────────────────────────────────┘

        [ Create pull request ]
```

操作顺序建议：

1. 先写好「范围 + 说明」（可先贴模板）  
2. 再拖入 PDF/Word  
3. （推荐）再粘贴「本次需求摘录」  
4. 点 Create pull request  

PR 后来补材料：Edit 描述，同样改这几项即可；`synchronize` / 编辑后再次触发审查时会用新描述。

## 系统侧怎么「读进来」（实现时按此做）

```text
Webhook: pull_request opened / synchronize
        ↓
读 pull_request.body（描述 Markdown）
        ↓
解析器抽出：
  - 范围、说明、摘录
  - 附件 URL（.pdf / .doc / .docx）
        ↓
若有附件 URL → 下载 → 抽全文 → 按「范围」切片
若有「本次需求摘录」→ 优先/合并进上下文
若都没有 → 降级：只做代码审查，并评论说明「未绑定 PRD」
        ↓
prompt = 需求片段 + 范围说明 + diff
        ↓
评论：业务符合 / 代码质量 / 缺口风险 / 分数
```

### 解析规则（建议写死，避免人乱写）

识别描述里这些键名（大小写不敏感、允许中英文）：

- `范围` / `Range` / `PRD范围`  
- `说明` / `Note` / `边界`  
- `本次需求摘录`  

附件：从 body 里抓链接，扩展名为 `.pdf` `.doc` `.docx` 的优先当作 PRD。  
若有多个附件：取**第一个** PDF/Word，或标题含 `PRD` 的那个（实现时再细化）。

## 附件下载要注意的坑（设计阶段先知晓）

GitHub 在描述里拖文件后，链接常为：

`https://github.com/user-attachments/...`

注意：

1. **人在网页上创建时最省事**——完全符合我们的产品路径。  
2. 服务器用 Token 去下载 `user-attachments` 时，个别情况可能遇到鉴权限制。  
3. 因此第一版做**双通道**，避免单点失败：  

| 通道 | 来源 | 作用 |
|------|------|------|
| A. 附件 URL | 拖拽上传 | 主路径：抽 PRD 全文再切片 |
| B. 本次需求摘录 | 描述里粘贴 | 兜底：不依赖下载也能比对 |
| C. 仓库内路径（可选） | 如 `docs/prd/xxx.pdf` | 私有仓更稳的备选 |

评论里可写清用了哪条通道，方便排错。

## 没有填模板时怎么办（策略）

| 情况 | 行为 |
|------|------|
| 完全没有需求依据 | 仅代码审查 + 评论提示「未绑定 PRD」 |
| 有范围/说明但无文件、无摘录 | 仅代码审查 + 提示「缺 PRD 正文」 |
| 有摘录、无文件 | **可以做业务比对**（摘录当需求） |
| 有文件、范围解析失败 | 用全文前 N 字或要求补范围；同时若有摘录则用摘录 |
| 有文件 + 范围 + 摘录 | 完整业务+代码审查（理想路径） |

第一版**不阻断合并**（只评论），制度严格后再考虑「无 PRD 则失败」。

## 和现有审查链路的关系

```text
现在：diff → 大模型 → 评论
以后：PRD 片段 + diff +（按需）全局读码 → 大模型 → 分段评论
```

Webhook、Token、部署方式都不变；多人协作成本 ≈ **多填几行描述 + 拖一个文件**。

## 关键：依赖 PRD 时，不能只看 diff

### 问题是什么？

只把 **diff** 和 PRD 比对，只能回答：

> 「这次改动的几行，看起来像不像需求？」

回答不了更重要的：

> 「PRD 要求的能力，仓库里是不是都落实了？有没有该改却没出现在 diff 里的缺口？」

| 风险类型 | 只看 diff | 需要看全局 |
|----------|-----------|------------|
| 改错了 / 改歪了 | 能发现 | — |
| **该改没改**（漏实现） | **很难发现**（diff 里根本没有） | 要读现有代码、搜调用链 |
| 旧逻辑仍挡着新需求 | 难 | 要对照相关模块 |
| 测试/配置/权限没跟上 | 常漏 | 要按清单去翻仓库 |

所以：**有 PRD 的审查 = 需求核对 + 代码质量**，信息源至少三类：

1. **PRD 片段**（要做什么）  
2. **本次 diff**（这次动了什么）  
3. **相关全局代码**（仓库里实际有没有、对不对）

### 目标流程（在原流程上加「读库」）

```text
GitHub PR 打开/更新
        ↓
① 拿到 diff
        ↓
② 解析 PRD 绑定（文件 / 范围 / 说明 / 摘录）
        ↓
③ 取出 PRD 对应片段
        ↓
④ 从 PRD 拆出「可核对清单」（验收点列表）
        ↓
⑤ 对照清单读全局代码（克隆 PR 头提交 → 搜索/读文件）
        ↓
⑥ 组上下文给大模型：
   - 需求片段 + 清单
   - diff
   - 全局探查得到的证据（文件摘录 /「未找到」）
        ↓
⑦ 输出：已覆盖 / 未落实 / 冲突 / 代码质量 / 分数
        ↓
⑧ 评论回 PR（业务符合 + 代码质量 两段）
```

### 推荐策略：有 PRD 就走「探查」，无 PRD 仍可只看 diff

| 场景 | 策略 | 说明 |
|------|------|------|
| **未绑定 PRD** | `diff_only` | 保持现在的便宜、快速代码审 |
| **已绑定 PRD** | `prd_agentic`（名称待定） | 克隆仓库 + 按清单读文件/搜索，再出结论 |
| 探查失败 | 降级 | 至少用「PRD + diff」出一版，并注明「未完成全局核对」 |

项目里已有 **`REVIEW_STRATEGY=agentic`**（本地克隆 + `read_file` / 命令探查）。  
PRD 模式应复用这套能力，而不是再发明一套；差别是：**探查目标由 PRD 清单驱动，而不是漫无目的翻代码**。

### 「全局」不是「整仓塞进模型」

不要把整个仓库扔进 prompt。正确做法是 **有目的的局部探查**：

```text
PRD 片段
  → 模型/规则拆成验收点（例：退款失败要写状态、要发通知、要幂等）
  → 对每个点：搜索关键词 / 读可疑文件 / 看 diff 是否覆盖
  → 只把「证据片段 + 未找到」喂回最终总结
```

这样既补上「漏改」，又控制 tokens / 费用 / 耗时。

### 评论里建议固定结构

1. **需求覆盖**：清单逐条 → 已落实 / 未落实 / 证据路径  
2. **Diff 质量**：本次改动本身的问题  
3. **缺口与风险**：该改没改、副作用  
4. **分数**：可拆「业务符合分 + 代码质量分」或总分 + 子分  

### 代价（设计时就要接受）

| 项 | diff_only | 带 PRD 的全局探查 |
|----|-----------|-------------------|
| 耗时 | 短 | 更长（克隆 + 多轮工具） |
| 费用 | 低 | 明显更高 |
| 漏检「没改」 | 弱 | 强得多 |

产品策略可以是：**只有填了需求依据的 PR 才启用全局探查**；日常小修仍可用纯 diff。

## 给产品/开发的一页话术

> 开 PR 时：  
> 1）把 PRD（PDF/Word）拖进描述下方附件区；  
> 2）按模板写「范围」和「说明」；  
> 3）最好再粘贴相关段落到「本次需求摘录」。  
> AI 会对照需求看：这次改动对不对、以及仓库里该落实的有没有漏掉（不只看 diff）。
