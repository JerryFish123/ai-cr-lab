# PRD 如何挂进 PR（已拍板）

> 本文已按 2026-07-23 讨论结论固化，作为实现依据。

## 结论（先看这个）

| 人要提供的东西 | 放哪 |
|----------------|------|
| PRD 文件（PDF / Word） | PR **Description** 下方附件区拖入 |
| 本期对照的章节号（如 `1.xx`、`2.xx`）及说明文字 | 同一份 **PR Description**（自然语言即可） |

创建 / 更新 PR 后：

1. GitHub 把附件变成描述里的链接  
2. Webhook 触发审查  
3. 服务读 **PR 描述全文** → 下载附件 → 抽文本 → 结合章节说明与仓库探查 → 出评论  

**不需要**另做上传网站；第一版用 GitHub 原生能力。

---

## 已拍板决策（实现必须遵守）

| # | 决策 |
|---|------|
| 1 | 材料写在 **PR Description**，不用评论区做主入口 |
| 2 | `1.xx` / `2.xx` = **PRD 内章节号**（不是开发自编验收清单） |
| 3 | **不强制机器可读模板**；agentic 模式下文字 + PRD 能让 AI 看懂即可 |
| 4 | 「有没有覆盖」= **agentic 读仓库取证**；未覆盖时给出**覆盖建议 / 修改方向** |
| 5 | **完成度用百分比**表达，不要求非常精准 |
| 6 | 每次 PR 最多两条评论逻辑：**① 代码质量**（始终）；**② 需求完成情况**（仅当判定要做 PRD 分析时触发）。无 PRD 分析 = 视为无需求技术改造，只出① |
| 7 | 有文件但抽不出文本 → 评论 **「PRD解析失败」并附原因**，便于排查 |
| 8 | 同一 `last_commit_id` 已审过 → **直接 skip**（保持现逻辑） |
| 9 | push + PR 各审一次 → **本期不管**，后续再优化 |

## 开发同学怎么填（建议写法，非强制格式）

在 **Add a description** 里用自然语言说明即可，例如：

```text
本期对照 PRD 第 3.2、3.5 节改造退款失败状态流转。
附件为完整 PRD（PDF/Word）。
不做发票与对账。
```

把 PDF/Word 拖到描述下方附件区即可。

> 说明：系统侧**不依赖**固定 Markdown 键名解析；但若描述完全看不出章节/附件意图，则**不触发**「需求完成情况」分析（见下节触发条件）。

---

## 何时触发「需求完成情况」

| 情况 | ① 代码质量 | ② 需求完成情况 |
|------|------------|----------------|
| 未加入 PRD / 描述里看不出要做需求对照 | ✅ | ❌ 不触发（无需求技术改造） |
| 有 PRD 附件（或明确可解析的 PRD 正文）+ 描述里有章节/范围意图 | ✅ | ✅ |
| 有文件、抽不出文本 | ✅ | ❌；另发/附带 **PRD解析失败 + 原因** |

判定原则（实现时）：

- **有 PRD 可分析材料**才走需求完成逻辑（附件抽文本成功，或描述中已有足够 PRD 正文）。  
- 仅代码重构、无 PRD → 只出「代码质量」。

---

## 评论排版

### 评论 1：代码质量（现有能力，始终可出）

沿用当前 agentic / 代码审查输出。

### 评论 2：需求完成情况（仅 PRD 分析触发时）

建议结构（AI 自由组织，但需包含）：

1. **本期对照章节**（从描述 + PRD 理解到的章节号）  
2. **逐章覆盖结论**（已覆盖 / 部分 / 未覆盖 + 仓库证据路径）  
3. **总体完成度**：约 `xx%`（不要求精确）  
4. **未覆盖时的覆盖建议 / 修改方向**（改哪类模块、补什么逻辑）

若解析失败：

```text
PRD解析失败
原因：……（如下载鉴权失败 / 空文本 / 不支持格式 / 超时等）
```

## 系统侧流程（实现按此做）

```text
Webhook: pull_request / push（本期双触发可并存）
        ↓
同 commit 已审过？ → skip
        ↓
① 代码质量审查（现有 agentic）
        ↓
是否具备 PRD 分析条件？
  否 → 结束（仅代码质量；无「需求完成情况」）
  是 → 下载附件 → 抽文本
        ↓
抽文本失败 → 评论「PRD解析失败」+ 原因；结束需求侧
        ↓
成功 → agentic：按章节意图读仓库取证
        ↓
输出「需求完成情况」：覆盖结论 + 约 xx% + 修改方向
```

### 信息源（有 PRD 时）

1. **PRD 文本**（按描述中的章节号理解范围；不强制机器切片键）  
2. **本次 diff**  
3. **相关全局代码**（agentic `read_file` / 搜索等）——用于发现「该改没改」

**全局 ≠ 整仓塞进模型**：仍由 agentic 按章节意图局部探查，只回收证据片段。

## 附件下载注意

GitHub 描述拖文件后链接常为 `https://github.com/user-attachments/...`。

| 风险 | 处理 |
|------|------|
| Token 下载鉴权失败 | 「PRD解析失败」+ 原因 |
| PDF/Word 抽空文本 | 同上 |
| 多附件 | 优先第一个 `.pdf/.doc/.docx`，或文件名含 `PRD` 的（实现细化） |

可选后续增强（非本期必须）：仓库内路径 `docs/prd/xxx.pdf` 作为备选通道。

---

## 边界策略汇总

| 情况 | 行为 |
|------|------|
| 无 PRD、纯技术改动 | 仅「代码质量」 |
| 有 PRD 且可抽文本 | 「代码质量」+「需求完成情况」 |
| 有文件抽不出文本 | 「代码质量」+「PRD解析失败」及原因 |
| 同一 commit 已审 | skip |
| push + PR 各一次 | 本期接受双触发 |

第一版**不阻断合并**（只评论）。

---

## 代价（接受）

有 PRD 时耗时/费用明显高于纯代码审（克隆 + 多轮工具 + 需求总结）。  
产品策略：**只有要做需求对照的 PR 才启用需求完成分析**。

---

## 给产品/开发的一页话术

> 开 PR 时：把 PRD（PDF/Word）拖进描述附件区，并在描述里写明对照的章节号（如 3.2、3.5）和本期边界。  
> AI 会先出「代码质量」；若成功解析 PRD，再出「需求完成情况」（覆盖结论、约 xx% 完成度、未覆盖时的修改方向）。  
> 没挂 PRD 的技术改造只审代码，不跑需求完成分析。
